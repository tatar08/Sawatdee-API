1. บทนำและแนวคิดหลัก

Sawatdee API ได้รับการออกแบบให้เป็นทางเลือกแทน Postman ที่ทำงานบนเบราว์เซอร์ของคุณโดยสมบูรณ์ โดยให้ความสำคัญกับความเป็นส่วนตัวของผู้ใช้งาน แตกต่างจากเครื่องมืออื่นที่ต้องส่งคำขอผ่านพร็อกซีเซิร์ฟเวอร์หรือบันทึกข้อมูลการทำงานของคุณไว้บนคลาวด์ Sawatdee API จะทำงานภายในเว็บเบราว์เซอร์ของคุณแบบ 100%

คุณสมบัติเด่น:

  • ไม่มีระบบหลังบ้าน (Zero Backend): ส่งคำขอตรงจากเบราว์เซอร์ของคุณไปยัง API ปลายทางผ่านฟังก์ชัน fetch() โดยไม่มีใครมาคั่นกลาง
  • ปลอดภัยและออฟไลน์ (Offline-First): คอลเลกชัน, ประวัติการส่งคำขอ, สภาพแวดล้อม และการตั้งค่าทั้งหมดจะถูกบันทึกไว้ในเบราว์เซอร์ของคุณโดยใช้ IndexedDB (ผ่าน Dexie) โทเค็นและคีย์ความลับของคุณจะไม่รั่วไหลออกไปภายนอก
  • ดีไซน์สวยงามสไตล์ Apple: ออกแบบด้วยเอฟเฟกต์กระจกฝ้า (Glassmorphic "Liquid Glass") โทนสีพาสเทลอ่อนโยน และภาพสัตว์นำโชคญี่ปุ่นที่เปลี่ยนได้ตามใจชอบ

2. CORS: ข้อจำกัดความปลอดภัยบนเบราว์เซอร์

เนื่องจาก Sawatdee API ทำงานบนเว็บเบราว์เซอร์ของคุณโดยตรง ระบบจึงต้องอยู่ภายใต้มาตรการความปลอดภัย Cross-Origin Resource Sharing (CORS) ของเบราว์เซอร์ ซึ่งเป็นมาตรการความปลอดภัยระดับสากล ไม่ใช่ข้อผิดพลาดของ Sawatdee API

สรุปความเข้าใจเรื่อง CORS:

  • API ที่รองรับ CORS: จะทำงานได้อย่างสมบูรณ์หากเซิร์ฟเวอร์ปลายทางส่งเฮดเดอร์ Access-Control-Allow-Origin: * เพื่ออนุญาตให้เบราว์เซอร์อ่านค่าได้
  • API ที่บล็อก CORS: หากเซิร์ฟเวอร์ไม่ส่งเฮดเดอร์ CORS กลับมา เบราว์เซอร์จะบล็อกผลลัพธ์ทันที Sawatdee API จะแสดง แผงอธิบาย CORS (CORS Explainer) เพื่อช่วยวินิจฉัยปัญหา
  • บล็อกเนื้อหาผสม (Mixed Content): เบราว์เซอร์จะบล็อกการเรียกใช้ API ที่เป็น HTTP (ไม่ปลอดภัย) จากหน้าเว็บที่เป็น HTTPS (ปลอดภัย) Sawatdee API จะตรวจพบและแสดงคำเตือนทันที
  • เฮดเดอร์ต้องห้าม (Forbidden Headers): เบราว์เซอร์จะบล็อกการแก้ไขเฮดเดอร์บางตัวเพื่อความปลอดภัย (เช่น Host, Origin, User-Agent, Cookie) Sawatdee API จะแสดงคำเตือนหากคุณพยายามแก้ไขคีย์เหล่านี้

3. การตั้งค่าและการรันระบบในเครื่อง (Local Dev)

หากต้องการรัน Sawatdee API ในเครื่องคอมพิวเตอร์ของคุณหรือนำไปโฮสต์เอง:

สิ่งที่ต้องเตรียม:

ตรวจสอบว่าเครื่องคอมพิวเตอร์ของคุณติดตั้ง Node.js และ npm หรือ pnpm เรียบร้อยแล้ว

คำสั่งที่ใช้งาน:

# ติดตั้งไลบรารีที่จำเป็น
pnpm install

# เริ่มทำงานในโหมดพัฒนา (Local Server)
pnpm dev

# รวมโค้ดสำหรับการนำไปใช้งานจริง (Build production)
pnpm build

ผลลัพธ์สำหรับการโฮสต์จริงจะอยู่ในโฟลเดอร์ dist/ ซึ่งประกอบไปด้วยไฟล์ HTML, CSS, JS สแตติกทั้งหมด

5. การสร้างและปรับแต่งคำขอ (Request Construction)

5.1 HTTP Methods

รองรับการส่งคำขอมาตรฐาน: GET, POST, PUT, PATCH, DELETE, HEAD, OPTIONS

5.2 การแทนที่ตัวแปร

คุณสามารถแทรกคีย์ตัวแปรที่ต้องการแทนค่าลงใน URL หรือช่องกรอกข้อมูลอื่นๆ ในรูปแบบปีกกาซ้อนสองชั้น เช่น {{host}} หรือ {{userId}}

5.3 ประเภทการยืนยันตัวตน (Authorization)

  • Bearer Token: เพิ่มส่วนหัว Authorization: Bearer <token> โดยอัตโนมัติ
  • Basic Auth: กรอก Username และ Password ซึ่งระบบจะทำการแปลงด้วยรูปแบบ Base64
  • API Key: ส่งพารามิเตอร์หรือข้อมูลส่วนหัว API Key ที่คุณสร้างขึ้น

หมายเหตุ: ระบบจะคอยอัปเดต Live Preview ของข้อมูลยืนยันตัวตนให้คุณตรวจสอบก่อนกดปุ่มส่งคำขอเสมอ

5.4 บอดี้ของคำขอ (Body)

กำหนดข้อมูลที่จะส่งไปได้ที่แถบ Body โดยเลือกโหมด JSON, Raw text, Form data หรือ URL-encoded

6. การวิเคราะห์ผลลัพธ์ตอบกลับและการแก้ไขปัญหา (Response Panel)

หลังส่งคำขอ แผงผลลัพธ์จะแสดงผลข้อมูลรายละเอียดทั้งหมด:

  • Pretty: แสดงข้อความตอบกลับในรูปแบบสวยงามจัดหมวดหมู่ โครงสร้างข้อมูลอ่านง่าย
  • Raw: แสดงรายละเอียดข้อความทั้งหมดไม่มีการตัดทอน
  • Preview: แสดงผลลัพธ์รูปภาพโดยตรง หรือโหลดหน้าเว็บจำลองผ่าน iframe ปลอดภัย
  • ค้นหาคำ: พิมพ์คีย์เวิร์ดในแถบค้นหาเพื่อค้นหาข้อมูลและแสดงจำนวนคำพร้อมไฮไลต์สี

7. ระบบสภาพแวดล้อมและตัวแปร (Environments & Variables)

สลับการส่งข้อมูลไปยั่งเซิร์ฟเวอร์ที่ต้องการได้รวดเร็วผ่านระบบสภาพแวดล้อม (Environments):

  1. คลิกเลือกเมนูสภาพแวดล้อมที่แถบด้านบน (หรือเปิดหน้าต่างตั้งค่า)
  2. สร้างสภาพแวดล้อมใหม่ เช่น "Dev" หรือ "Staging" พร้อมระบุข้อมูลตัวแปร
  3. เลือกสภาพแวดล้อมที่ใช้งาน placeholders เช่น {{userId}} ในช่องกรอกข้อมูลจะได้รับการแทนค่าอัตโนมัติ

8. การนำเข้าและการส่งออกข้อมูล (Migration)

ย้ายคอลเลกชันและข้อมูลเพื่อนำไปทดสอบบนระบบอื่นได้อย่างยืดหยุ่น:

  • การนำเข้าข้อมูล: รองรับไฟล์ Postman Collections v2.1, Postman Environments หรือไฟล์ Native ของ Sawatdee API ระบบจะตรวจจับประเถทของข้อมูลโดยอัตโนมัติ
  • แก้ปัญหาชื่อซ้ำ: ระบบจะเติมคำต่อท้ายว่า (imported) เพื่อไม่ให้ไฟล์เดิมถูกเขียนทับ
  • การส่งออกข้อมูล: ส่งออกแยกเป็นคอลเลกชันเดี่ยว หรือแบ็กอัปข้อมูล Sawatdee API ทั้งหมด

9. ฟังก์ชันขั้นสูง: การรันคอลเลกชันด้วยไฟล์ข้อมูล (Collection Runner)

Collection Runner ช่วยคุณทดสอบส่งคำขอ API แบบวนลูปตามรายการข้อมูลจากไฟล์ CSV หรือ JSON โดยสามารถตั้งค่าพารามิเตอร์การทำงานเพิ่มเติมได้ดังนี้:

  • จำนวนรอบการรัน (Iterations) และดีเลย์ (Delay): กำหนดจำนวนรอบที่ต้องการรันคำขอวนซ้ำ และกำหนดเวลาดีเลย์ในการรอระหว่างส่งคำขอ (หน่วยเป็นมิลลิวินาที)
  • สภาพแวดล้อมที่ใช้งาน (Active Environment): เลือกหรือสลับ Environment ได้โดยตรงในมอดัลการรันผ่าน Dropdown
  • การเลือกคำขอ (Request Checklist): ติ๊กเลือกหรือยกเลิกรายการที่จะส่งคำขอรันได้ตามต้องการ หรือกด "เลือกทั้งหมด" และ "ยกเลิกการเลือกทั้งหมด"

ขั้นตอนการใช้งานร่วมกับไฟล์ข้อมูล:

  1. คลิกเมนูคอลเลกชันด้านซ้าย แล้วเลือก Run Collection
  2. อัปโหลดไฟล์ข้อมูลจำลอง CSV หรือ JSON ใต้ช่อง Data File
  3. ใช้ชื่อหัวคอลัมน์เขียนเป็นตัวแปร เช่น {{username}} ในข้อมูลคำขอของคุณ
  4. ปรับตั้งค่าดีเลย์, รอบการรัน, เลือก Environment และเลือกเฉพาะคำขอที่ต้องการทดสอบ
  5. กดเริ่มรัน ระบบจะทำคำขอวนลูปและแสดงความคืบหน้าให้ทราบแบบเรียลไทม์

การส่งออกรายงานผลลัพธ์ (Export Results):

หลังการรันเสร็จสิ้น คุณสามารถคลิกปุ่ม ส่งออกผลลัพธ์ (Export Results) เพื่อดาวน์โหลดข้อมูลสรุปผลการรันเป็นไฟล์ JSON เก็บไว้บันทึกเป็นรายงานของแต่ละรอบได้อย่างละเอียด

10. ธีม, ภาพพื้นหลัง, และสัตว์นำโชค (Aesthetics & Themes)

Sawatdee API มีธีมและภาพวาดสไตล์พาสเทลน่ารักหลากหลาย เช่น ชมพูคลาสสิก, ชิบะอินุ, แมวกวักนำโชค, กระต่ายชมจันทร์, แพนด้าชาเขียว และ ช้างไทย

  • โหมดมืด (Dark Mode): ปรับระดับความจางของพื้นหลังและเปลี่ยนธีมเป็นสีโทนมืด เพื่อถนอมสายตาของคุณ
  • ลดความโปร่งใส: ปรับเลเยอร์กระจกฝ้าเป็นสีทึบ เพื่อปรับปรุงคอนทราสต์และความชัดเจนในการมองเห็น

11. ระบบความปลอดภัยและการใช้งานภายในองค์กร (Security & Enterprise Features)

เพื่อสนับสนุนการนำ Sawatdee API ไปปรับใช้งานอย่างปลอดภัยภายในเครือข่ายขององค์กร ระบบได้เพิ่มมาตรการความปลอดภัยและกลไกการจำกัดสิทธิ์เข้าถึงดังต่อไปนี้:

11.1 ระบบล็อกแอปด้วยข้อมูลประจำตัว (Client-Side Username & PIN Lock)

เพื่อป้องกันไม่ให้ผู้อื่นเข้าถึงประวัติคำขอ ตัวแปรสภาพแวดล้อม หรือข้อมูลคีย์ลับที่บันทึกอยู่ใน IndexedDB บนเบราว์เซอร์เดียวกัน:

  • โหมดตั้งค่าเริ่มต้น (Setup Mode): หากเปิดแอปครั้งแรกและยังไม่มีข้อมูลสิทธิ์เข้าใช้งาน ระบบจะบังคับให้ตั้ง ชื่อผู้ใช้งาน (Username) (ความยาวอย่างน้อย 3 ตัวอักษร) พร้อมด้วย รหัส PIN ความยาว 6-8 หลัก และทำการยืนยันรหัส PIN ให้ตรงกันอีกครั้ง
  • หน้าจอล็อก (Lock Screen): เมื่อโหลดหน้าเว็บหรือกดรีเฟรช แอปจะแสดงหน้าจอล็อกเพื่อความปลอดภัย โดยผู้ใช้งานจะต้องป้อนทั้ง Username และ รหัส PIN ให้ถูกต้อง จึงจะผ่านเข้าสู่พื้นที่ทำงานหลักได้ ระหว่างล็อกระบบจะปิดกั้นหน้าจอและคีย์บอร์ดลัดทั้งหมด
  • การจัดเก็บข้อมูลแบบ Local: ชื่อผู้ใช้จะถูกเก็บเป็นข้อความธรรมดา (username) ใน IndexedDB บนเครื่องของผู้ใช้ ส่วนรหัส PIN จะถูกเข้ารหัสแฮชด้วยอัลกอริทึม SHA-256 ผ่าน Web Crypto API ในเบราว์เซอร์ก่อนทำการบันทึกเป็น pinHash โดยไม่มีการส่งข้อมูลเหล่านี้ออกนอกเครื่องไปยังเซิร์ฟเวอร์ใด ๆ
  • ระบบป้องกันและล้างข้อมูลอัตโนมัติ (Auto-Wipe): หากป้อนข้อมูลผู้ใช้หรือรหัส PIN ผิดติดต่อกันรวมเป็นจำนวน 5 ครั้ง ระบบจะทำการลบข้อมูลประวัติ คอลเลกชัน สภาพแวดล้อม และการตั้งค่าทั้งหมดที่บันทึกอยู่ในเบราว์เซอร์นี้ทันทีเพื่อรักษาความปลอดภัย จากนั้นจะรีเซ็ตแอปกลับสู่หน้าตั้งค่าเริ่มต้นใหม่
  • ปุ่มรีเซ็ตแบบแมนนวล (Forgot details): หากจำข้อมูลไม่ได้ พนักงานสามารถคลิก Forgot details? เพื่อยืนยันคำสั่งล้างข้อมูลในเครื่องทั้งหมดและเริ่มกระบวนการตั้งค่า Username และ PIN ใหม่ได้ด้วยตัวเอง

11.2 ระบบรักษาความปลอดภัยของ CORS Proxy (api/proxy.ts)

เมื่อรันระบบ CORS Proxy เพื่อยิง Request ไปยังเซิร์ฟเวอร์ที่ไม่รองรับ CORS หรืออยู่ในเครือข่ายเฉพาะ Proxy มีระบบตรวจสอบความปลอดภัยเสริมดังนี้:

  • การป้องกัน SSRF (Server-Side Request Forgery): ตัว Proxy จะทำการปฏิเสธคำขอที่พยายามเชื่อมต่อกลับมายัง Localhost (localhost, 127.0.0.1, ::1), ที่อยู่ IP ภายในองค์กร (Private IP ได้แก่ 10.x.x.x, 192.168.x.x, 172.16.x.x - 172.31.x.x) หรือ API เมทาดาตาของระบบคลาวด์ (169.254.169.254)
  • กำหนด Whitelist โดเมนที่อนุญาต (ALLOWED_DOMAINS): สามารถจำกัดให้ Proxy ยิง Request ไปยังโดเมนเฉพาะที่ได้รับการยอมรับผ่านตัวแปรสภาพแวดล้อม ALLOWED_DOMAINS (คั่นด้วยเครื่องหมายจุลภาค ,)
  • การยืนยันตัวตนผ่าน Shared Secret (PROXY_SECRET): หากกำหนดตัวแปรสภาพแวดล้อม PROXY_SECRET ในระดับเซิร์ฟเวอร์ ทุกคำขอที่ส่งมาที่ Proxy จะต้องแนบเฮดเดอร์ x-proxy-secret ที่มีค่าตรงกัน มิฉะนั้นจะได้รับ HTTP 401 Unauthorized
  • การยืนยันตัวตนผ่าน JWT (PROXY_JWT_SECRET): หากต้องการตรวจสอบสิทธิ์พนักงานด้วยระบบ Identity Provider (IdP) ขององค์กร สามารถตั้งค่า PROXY_JWT_SECRET เพื่อให้ตัว Proxy ตรวจสอบลายเซ็น (HMAC SHA-256 Signature) ของ JWT Token ที่แนบมาในส่วนหัว Authorization: Bearer <token> ได้
