## 1. บทนำและแนวคิดหลัก

Sawatdee API ได้รับการออกแบบให้เป็นทางเลือกแทน Postman ที่ทำงานบนเบราว์เซอร์ของคุณโดยสมบูรณ์ โดยให้ความสำคัญกับความเป็นส่วนตัวของผู้ใช้งาน แตกต่างจากเครื่องมืออื่นที่ต้องส่งคำขอผ่านพร็อกซีเซิร์ฟเวอร์หรือบันทึกข้อมูลการทำงานของคุณไว้บนคลาวด์ **Sawatdee API จะทำงานภายในเว็บเบราว์เซอร์ของคุณแบบ 100%**

### คุณสมบัติเด่น:

- **ไม่มีระบบหลังบ้าน (Zero Backend)**: ส่งคำขอตรงจากเบราว์เซอร์ของคุณไปยัง API ปลายทางผ่านฟังก์ชัน `fetch()` โดยไม่มีใครมาคั่นกลาง
- **ปลอดภัยและออฟไลน์ (Offline-First)**: คอลเลกชัน, ประวัติการส่งคำขอ, สภาพแวดล้อม และการตั้งค่าทั้งหมดจะถูกบันทึกไว้ในเบราว์เซอร์ของคุณโดยใช้ **IndexedDB** (ผ่าน Dexie) โทเค็นและคีย์ความลับของคุณจะไม่รั่วไหลออกไปภายนอก
- **ดีไซน์สวยงามสไตล์ Apple**: ออกแบบด้วยเอฟเฟกต์กระจกฝ้า (Glassmorphic "Liquid Glass") โทนสีพาสเทลอ่อนโยน และภาพสัตว์นำโชคญี่ปุ่นที่เปลี่ยนได้ตามใจชอบ

## 2. CORS: ข้อจำกัดความปลอดภัยบนเบราว์เซอร์

เนื่องจาก Sawatdee API ทำงานบนเว็บเบราว์เซอร์ของคุณโดยตรง **ระบบจึงต้องอยู่ภายใต้มาตรการความปลอดภัย Cross-Origin Resource Sharing (CORS) ของเบราว์เซอร์** ซึ่งเป็นมาตรการความปลอดภัยระดับสากล ไม่ใช่ข้อผิดพลาดของ Sawatdee API

### สรุปความเข้าใจเรื่อง CORS:

- **API ที่รองรับ CORS**: จะทำงานได้อย่างสมบูรณ์หากเซิร์ฟเวอร์ปลายทางส่งเฮดเดอร์ `Access-Control-Allow-Origin: *` หรือส่งโดเมนของหน้าเว็บของคุณกลับมา เพื่ออนุญาตให้เบราว์เซอร์อ่านค่าได้
- **API ที่บล็อก CORS**: หากเซิร์ฟเวอร์ไม่ส่งเฮดเดอร์ CORS กลับมา เบราว์เซอร์จะบล็อกผลลัพธ์ทันที Sawatdee API จะเปิดหน้าต่าง **อธิบายสาเหตุ CORS (CORS Explainer)** โดยเฉพาะเพื่อช่วยวินิจฉัยปัญหา
- **บล็อกเนื้อหาผสม (Mixed Content)**: เบราว์เซอร์จะบล็อกการเรียกใช้ API ที่เป็น HTTP (ไม่ปลอดภัย) จากหน้าเว็บที่เป็น HTTPS (ปลอดภัย) Sawatdee API จะตรวจพบและแสดงคำเตือนทันที
- **เฮดเดอร์ต้องห้าม (Forbidden Headers)**: เบราว์เซอร์จะบล็อกการแก้ไขเฮดเดอร์บางตัวเพื่อความปลอดภัย (เช่น `Host`, `Origin`, `User-Agent`, `Cookie`, `Content-Length`) Sawatdee API จะแสดงคำเตือนหากคุณพยายามแก้ไขคีย์เหล่านี้

## 3. การตั้งค่าและการรันระบบในเครื่อง (Local Dev)

หากต้องการรัน Sawatdee API ในเครื่องคอมพิวเตอร์ของคุณหรือนำไปโฮสต์เอง:

### สิ่งที่ต้องเตรียม:

- **Node.js** (แนะนำเวอร์ชัน 18 ขึ้นไป)
- **npm** หรือ **pnpm**

### วิธีการติดตั้ง:

1. ดาวน์โหลดหรือคัดลอกโฟลเดอร์โครงการ (Project directory) ลงในเครื่องของคุณ
2. เปิดโปรแกรม Terminal/Command Prompt ในโฟลเดอร์นั้น
3. ติดตั้งไลบรารีที่จำเป็น:
   ```bash
   pnpm install  # หรือ npm install
   ```

### เริ่มทำงานในโหมดพัฒนา (Development Mode):

รันคำสั่งด้านล่างเพื่อเปิด Vite Dev Server พร้อมระบบโหลดซ้ำอัตโนมัติเมื่อแก้ไขโค้ด (HMR):

```bash
pnpm dev      # หรือ npm run dev
```

จากนั้นเปิดเบราว์เซอร์ไปที่ `http://localhost:5173/`

### การสร้างไฟล์สำหรับนำไปใช้งานจริง (Production Build):

รวมโค้ดและย่อขนาดไฟล์ให้เหมาะกับการรันบนเว็บเซิร์ฟเวอร์:

```bash
pnpm build    # หรือ npm run build
```

ผลลัพธ์ทั้งหมดจะอยู่ในโฟลเดอร์ `dist/` ซึ่งคุณสามารถนำไปอัปโหลดขึ้นเว็บโฮสติ้งแบบคงที่ (Static Hosting) ใดๆ ก็ได้ เช่น Azure Static Web Apps, GitHub Pages, Netlify หรือ Vercel

### ส่วนประกอบต่างๆ บนหน้าจอ:

## 5. การสร้างและปรับแต่งคำขอ (Request Construction)

### 5.1 เมธอดที่รองรับ (HTTP Methods)

รองรับการทดสอบเมธอดมาตรฐาน ได้แก่ `GET`, `POST`, `PUT`, `PATCH`, `DELETE`, `HEAD`, `OPTIONS`

### 5.2 การใช้งานตัวแปร (Variable Interpolation)

คุณสามารถใส่ตัวแปรไว้ใน URL, ค่าเฮดเดอร์ หรือพารามิเตอร์ต่างๆ โดยใช้ปีกกาซ้อนกันสองชั้น เช่น `{{userId}}` ซึ่งระบบจะแทนค่าให้โดยอัตโนมัติจากค่าในสภาพแวดล้อมปัจจุบันหรือค่าระดับคอลเลกชัน ตัวแปรใดที่ยังไม่ตั้งค่าจะถูกแจ้งเตือนในแถบใต้ URL

### 5.3 รูปแบบการยืนยันตัวตน (Authorization Modes)

สามารถตั้งค่าการยืนยันตัวตนแยกทีละคำขอได้ โดยระบบจะเพิ่มเฮดเดอร์ที่ถูกต้องให้เองเมื่อกดส่ง:

- **None**: ไม่ส่งข้อมูลยืนยันตัวตนใดๆ
- **Bearer Token**: กรอกโทเค็นระบบจะแปลงเป็นเฮดเดอร์ `Authorization: Bearer <token>`
- **Basic Auth**: กรอก Username และ Password ระบบจะแปลงให้อยู่ในรูป Base64 และสร้างเป็น `Authorization: Basic <base64>`
- **API Key**: กำหนดชื่อคีย์, ค่าคีย์ และเลือกว่าจะส่งใน "เฮดเดอร์" หรือ "พารามิเตอร์คิวรี"

> [!NOTE]  
> ใต้แท็บยืนยันตัวตน จะมีส่วน **แสดงตัวอย่างแบบเรียลไทม์ (Live Preview)** เพื่อบอกว่าระบบจะนำค่าใดไปแทรกในคำขอจริง และจะเตือนคุณหากคุณมีการพิมพ์เฮดเดอร์ซ้ำกับข้อมูลยืนยันตัวตน

### 5.4 ส่วนข้อมูลคำขอ (Request Body)

ปรับแต่งข้อมูลที่จะส่งไปในคำขอได้ที่แท็บ **บอดี้ (Body)**:

- **None**: ไม่ส่งข้อมูลใดๆ ไป (เช่น คำขอ GET)
- **JSON**: เขียน JSON Payload โดยมีระบบช่วยไฮไลต์สีและตรวจไวยากรณ์ด้วย CodeMirror Editor
- **Raw**: ส่งข้อมูลดิบพร้อมเลือกไฮไลต์ภาษาปลายทางได้ (`text`, `xml`, `html`)
- **Form data / URL encoded**: ปรับแต่งค่าแบบตารางคีย์-ค่า รองรับการใช้ตัวแปร `{{variables}}`

## 6. การวิเคราะห์ผลลัพธ์ตอบกลับและการแก้ไขปัญหา (Response Panel)

เมื่อได้รับข้อมูลกลับมาจากเซิร์ฟเวอร์ แผงตอบกลับจะมีเครื่องมือช่วยวิเคราะห์ดังนี้:

### 6.1 โหมดแสดงผลบอดี้ (Body Sub-modes)

- **Pretty**: แสดงผล JSON/XML แบบจัดรูปเลย์เอาต์สวยงามพร้อมระบบไฮไลต์สี
- **Raw**: แสดงข้อความดิบทั้งหมดที่ได้รับจากเซิร์ฟเวอร์
- **Preview (แสดงตัวอย่าง)**:
  - **รูปภาพ**: แสดงรูปภาพผลลัพธ์ (`image/png`, `image/jpeg` ฯลฯ) บนเว็บเบราว์เซอร์ได้ทันทีโดยใช้ secure blob URL
  - **HTML**: แสดงตัวอย่างหน้าเว็บ HTML ภายในเฟรมจำลองพิเศษ (`<iframe>`) ที่ปิดกั้นการทำงานของสคริปต์ที่เป็นอันตราย (Sandboxed) เพื่อความปลอดภัย

### 6.2 การค้นหาและคัดกรองคำ

มีแถบค้นหาข้อความภายในหน้าแสดงผลตอบกลับทั้งโหมด Pretty และ Raw พร้อมไฮไลต์สีพื้นหลังคำที่ตรงกันและนับจำนวนคำให้โดยอัตโนมัติ

### 6.3 ระบบแจ้งเตือนข้อผิดพลาด CORS

หากการส่งคำขอล้มเหลวเนื่องจากมาตรการความปลอดภัยเบราว์เซอร์ Sawatdee API จะเปิดแผงอธิบายอย่างละเอียด:

```
+----------------------------------------------------------------------------------+
| ⚠️ คำขอถูกบล็อกหรือไม่สามารถเข้าถึงได้                                                |
| Sawatdee API ทำงานบนเบราว์เซอร์ของคุณโดยสมบูรณ์ ข้อผิดพลาดของเครือข่ายนี้มักหมายความว่า:    |
| 1. API ปลายทางไม่อนุญาตให้เข้าถึงจากเบราว์เซอร์ข้ามโดเมน (CORS)                    |
| 2. URL ไม่ถูกต้อง หรือเซิร์ฟเวอร์ปลายทางปิดอยู่                                      |
|                                                                                  |
| รายการตรวจสอบ:                                                                    |
|  ✓ ตรวจสอบว่า URL ถูกต้องและเซิร์ฟเวอร์เปิดอยู่หรือไม่?                              |
|  ✓ เซิร์ฟเวอร์ส่ง Access-Control-Allow-Origin หรือไม่?                              |
|  ⚠️ หน้าเว็บ HTTPS ไม่สามารถเรียกใช้ API ที่เป็น HTTP ได้ (เนื้อหาผสมโดนบล็อก)         |
|                                                                                  |
| หมายเหตุ: นี่คือระบบความปลอดภัยของเบราว์เซอร์ ไม่ใช่ข้อผิดพลาดของ Sawatdee API            |
+----------------------------------------------------------------------------------+
```

## 7. ระบบสภาพแวดล้อมและตัวแปร (Environments & Variables)

ระบบช่วยให้คุณสลับเป้าหมายของการส่งคำขอได้ง่ายขึ้น (เช่น ย้ายระหว่างเครื่อง Local ของคุณและเครื่องเซิร์ฟเวอร์จริง)

1. คลิกที่ปุ่ม **⚙ ตั้งค่า** บนแถบเมนูด้านบน แล้วไปที่เมนู **สภาพแวดล้อม (Environments)** (หรือคลิกที่ปุ่มเลือกสภาพแวดล้อมด้านบนโดยตรง)
2. สร้างสภาพแวดล้อมใหม่พร้อมกำหนดคีย์และค่าของตัวแปรในตาราง:
   ```
   ชื่อสภาพแวดล้อม: Local Dev
   +------------------------------+
   | คีย์ (Key) | ค่า (Value)      |
   +-----------+------------------+
   | host      | localhost:3000   |
   | userId    | 420              |
   +------------------------------+
   ```
3. ปิดหน้าต่างแล้วคลิกเลือก **Local Dev** ที่เมนูด้านบน
4. ในช่องกรอก URL ให้เขียน `{{host}}/users/{{userId}}` ระบบจะแทนค่าเป็น `localhost:3000/users/420` โดยอัตโนมัติเมื่อส่งคำขอ

## 8. การนำเข้าและการส่งออกข้อมูล (Migration)

ช่วยให้การย้ายข้อมูลเข้า-ออกเป็นเรื่องง่ายและไม่ต้องกรอกข้อมูลใหม่ทั้งหมด

### 8.1 ฟอร์แมตที่รองรับในการนำเข้า (Import)

ระบบจะตรวจจับประเภทของไฟล์โดยอัตโนมัติเมื่อกดเลือกไฟล์:

- **Postman Collections v2.1** (นำเข้าคอลเลกชันคำขอและพารามิเตอร์ทั้งหมด)
- **Postman Environments** (นำเข้าตัวแปรสภาพแวดล้อม)
- **Sawatdee API Native Bundles** (กู้คืนฐานข้อมูลการทำงานทั้งหมดของ Sawatdee API)

> [!IMPORTANT]  
> **การแก้ไขปัญหาชื่อซ้ำ**: ระบบ Sawatdee API จะไม่เขียนทับข้อมูลเดิมที่มีอยู่ หากมีรายการชื่อซ้ำกัน รายการที่นำเข้ามาใหม่จะถูกใส่คำต่อท้ายว่า `(imported)` ให้โดยอัตโนมัติ

### 8.2 ฟอร์แมตในการส่งออก (Export)

- **Postman Collection v2.1**: ส่งออกคำขอเฉพาะคอลเลกชันเพื่อนำไปใช้ต่อใน Postman
- **Sawatdee API Native**: ส่งออกไฟล์แบ็กอัปประวัติการทำงานและคอลเลกชันเพื่อบันทึกไว้ในคอมพิวเตอร์ของคุณ

## 9. ฟังก์ชันขั้นสูง: การรันคอลเลกชันด้วยไฟล์ข้อมูล (Collection Runner)

ฟีเจอร์เด่นอย่างหนึ่งของระบบคือการทดสอบ API ทีละหลายๆ รายการติดต่อกันด้วยไฟล์ข้อมูลจำลอง เช่น CSV หรือ JSON เพื่อใช้ทดสอบประสิทธิภาพหรือทำงานซ้ำๆ ได้อย่างง่ายดาย

### 9.1 พารามิเตอร์การตั้งค่าการรัน (Run Configurations)

- **จำนวนรอบการรัน (Iterations)**: กำหนดจำนวนรอบที่ต้องการให้ระบบทำการส่งคำขอซ้ำ
  - หากอัปโหลดไฟล์ข้อมูล ระบบจะตั้งค่าเริ่มต้นตามจำนวนแถวข้อมูลที่ตรวจพบโดยอัตโนมัติ
  - หากกำหนดจำนวนรอบมากกว่าจำนวนแถวข้อมูล ระบบจะทำการรันวนลูปแถวข้อมูลซ้ำตามลำดับ
  - หากไม่มีไฟล์ข้อมูล ระบบจะรันตามค่าที่ตั้งไว้ (เริ่มต้นที่ `1` รอบ)
- **ดีเลย์ (Delay)**: กำหนดระยะเวลาหน่วงก่อนส่งคำขอถัดไป มีหน่วยเป็นมิลลิวินาที (ms) เช่น `1000 ms` เท่ากับ 1 วินาที เหมาะสำหรับการจำกัดความถี่การส่งคำขอเพื่อไม่ให้เซิร์ฟเวอร์รับภาระหนักเกินไป
- **สภาพแวดล้อมที่ใช้งาน (Active Environment)**: สลับเปลี่ยน Environment ในหน้าต่างมอดัลการรันได้ทันทีจากเมนู Dropdown ทำให้การแทนค่าตัวแปรใน Requests อ้างอิงตาม Environment ที่เลือก
- **การเลือกคำขอ (Request Checklist)**: มีปุ่ม Checkbox ให้คุณเลือก/ยกเลิกคำขอที่จะนำมารันในคอลเลกชันได้ทีละรายการ พร้อมปุ่ม **เลือกทั้งหมด** และ **ยกเลิกการเลือกทั้งหมด** เพื่อความรวดเร็ว

### 9.2 ขั้นตอนการรันไฟล์ข้อมูล (CSV / JSON)

1. คลิกเมนูย่อยถัดจากชื่อคอลเลกชันในแถบ Sidebar แล้วเลือก **Run Collection**
2. ในส่วน **Data File** ให้ทำการอัปโหลดไฟล์ **CSV** (ที่มีแถวหัวคอลัมน์) หรือไฟล์ **JSON Array** ที่มีโครงสร้างแถวข้อมูล:
   ```csv
   id,username,role
   101,alice,admin
   102,bob,user
   103,charlie,moderator
   ```
3. นำคีย์ในไฟล์ดังกล่าวมาเขียนในรูปตัวแปร เช่น `{{id}}`, `{{username}}`, `{{role}}` ในส่วนต่างๆ ของคำขอในคอลเลกชันนั้น
4. ปรับตั้งค่า **Delay**, **Iterations**, เลือก **Environment** และ **Requests** ที่ต้องการใช้งาน
5. กดปุ่ม **เริ่มรันคอลเลกชัน (Run Collection)** ระบบจะทำการส่งคำขอวนลูปทีละแถวข้อมูลให้โดยอัตโนมัติ
6. แผงควบคุมจะแสดงผลการทำงานแบบสดๆ:

```
+--------------------------------------------------------------------------------------+
| รันคอลเลกชัน: Admin Suite                                                             |
| สถานะ: เสร็จสิ้น   •   ผ่าน (Passed): 6   •   ล้มเหลว: 0   •   เวลาเฉลี่ย: 45 ms        |
+--------------------------------------------------------------------------------------+
| รอบที่        | คำขอ                  | สถานะ     | เวลา   | การทำงาน                    |
+-------------+-----------------------+----------+--------+----------------------------+
| [1]         | Get Profile (alice)   | 200 OK   | 40 ms  | [ดูรายละเอียด]               |
| [1]         | Update Role (admin)   | 204 OK   | 48 ms  | [ดูรายละเอียด]               |
| [2]         | Get Profile (bob)     | 200 OK   | 38 ms  | [ดูรายละเอียด]               |
| [2]         | Update Role (user)    | 204 OK   | 52 ms  | [ดูรายละเอียด]               |
+--------------------------------------------------------------------------------------+
```

คุณสามารถกดตรวจสอบรายละเอียดการส่งคำขอและผลลัพธ์ของแต่ละรอบย้อนหลังได้ หรือกดหยุดการรันได้ทุกเมื่อ

### 9.3 การส่งออกรายงานการรัน (Export Results)

เมื่อการรันเสร็จสิ้น คุณสามารถคลิกปุ่ม **ส่งออกผลลัพธ์ (Export Results)** เพื่อดาวน์โหลดบันทึกรายงานเก็บไว้ในเครื่องเป็นไฟล์ JSON ซึ่งรายงานจะประกอบด้วย:
- ข้อมูลคอลเลกชันและวันเวลาที่ทำการรัน
- ชื่อ Environment ที่ใช้รันคำขอ
- สรุปสถิติ (จำนวนคำขอทั้งหมด, ผ่าน, ล้มเหลว, และระยะเวลาเฉลี่ย)
- บันทึกการตอบกลับของแต่ละรอบอย่างละเอียด (URL, สถานะ, ระยะเวลา, ข้อมูลตอบกลับ, และเฮดเดอร์)

## 10. ธีม ภาพพื้นหลัง และสัตว์นำโชค (Aesthetics & Themes)

Sawatdee API มาพร้อมกับหน้าตาที่น่ารัก สวยงาม ด้วยตัวการ์ตูนสัตว์นำโชคญี่ปุ่นและสีสันพาสเทลที่อบอุ่น:

| ชื่อธีม (Theme)     | โทนสีหลัก                    | ภาพสัตว์นำโชคประกอบ             |
| ------------------- | ---------------------------- | ------------------------------- |
| **ชมพูคลาสสิก**     | ชมพูพาสเทลอ่อนและกระจกฝ้า    | โลโก้ Sawatdee API ดั้งเดิม     |
| **ชิบะอินุ**        | ส้มอิฐและเหลืองทองอบอุ่น     | น้องหมาสุนัขพันธุ์ชิบะอินุ      |
| **แมวกวักนำโชค**    | ครีมงาช้าง ขอบทองและแดง      | แมวกวัก (Maneki-Neko) กวักโชคดี |
| **กระต่ายชมจันทร์** | ม่วงทไวไลท์และลาเวนเดอร์อ่อน | กระต่ายใต้แสงจันทร์ (Usagi)     |
| **แพนด้าชาเขียว**   | เขียวมิ้นต์และโทนป่าไผ่เข้ม  | หมีแพนด้ากินกิ่งไม้ไผ่          |
| **ช้างไทย**         | ฟ้าน้ำทะเลและสีครามรอยัลบลู  | ช้างไทยตกแต่งลวดลายงดงาม        |

### การตั้งค่าการแสดงผลเพิ่มเติม:

- **โหมดมืด (Dark / Night Mode)**: รองรับการแสดงผลสำหรับทุกสัตว์นำโชค โดยเลเยอร์กระจกฝ้าจะเปลี่ยนสภาพเป็นโทนสีมืดกึ่งโปร่งใส (`rgba(21, 14, 18, 0.45)`) ช่วยถนอมสายตาเวลากลางคืน สามารถสลับได้ผ่านปุ่ม 🌓 บนแถบด้านบน
- **ลดความโปร่งใส (Reduce Transparency)**: สลับพื้นผิวกระจกฝ้าให้กลายเป็นแผงทึบแสง ช่วยปรับปรุงความชัดเจนของข้อความ (Contrast) สำหรับผู้มีปัญหาทางสายตา หรือช่วยเพิ่มประสิทธิภาพเครื่องคอมพิวเตอร์ที่ช้า
- **ความจางของรูปพื้นหลัง**: ในโหมดมืด รูปสัตว์นำโชคจะจางลงโดยอัตโนมัติ เพื่อไม่ให้รบกวนสายตาขณะอ่านข้อความสีขาว

## 11. ระบบความปลอดภัยและการใช้งานภายในองค์กร (Security & Enterprise Features)

เพื่อสนับสนุนการนำ Sawatdee API ไปปรับใช้งานอย่างปลอดภัยภายในเครือข่ายขององค์กร ระบบได้เพิ่มมาตรการความปลอดภัยและกลไกการจำกัดสิทธิ์เข้าถึงดังต่อไปนี้:

### 11.1 ระบบล็อกแอปด้วยข้อมูลประจำตัว (Client-Side Username & PIN Lock)
เพื่อป้องกันไม่ให้ผู้อื่นเข้าถึงประวัติคำขอ ตัวแปรสภาพแวดล้อม หรือข้อมูลคีย์ลับที่บันทึกอยู่ใน IndexedDB บนเบราว์เซอร์เดียวกัน:
- **โหมดตั้งค่าเริ่มต้น (Setup Mode)**: หากเปิดแอปครั้งแรกและยังไม่มีข้อมูลสิทธิ์เข้าใช้งาน ระบบจะบังคับให้ตั้ง **ชื่อผู้ใช้งาน (Username)** (ความยาวอย่างน้อย 3 ตัวอักษร) พร้อมด้วย **รหัส PIN** ความยาว 6-8 หลัก และทำการยืนยันรหัส PIN ให้ตรงกันอีกครั้ง
- **หน้าจอล็อก (Lock Screen)**: เมื่อโหลดหน้าเว็บหรือกดรีเฟรช แอปจะแสดงหน้าจอล็อกเพื่อความปลอดภัย โดยผู้ใช้งานจะต้องป้อนทั้ง **Username** และ **รหัส PIN** ให้ถูกต้อง จึงจะผ่านเข้าสู่พื้นที่ทำงานหลักได้ ระหว่างล็อกระบบจะปิดกั้นหน้าจอและคีย์บอร์ดลัดทั้งหมด
- **การจัดเก็บข้อมูลแบบ Local**: ชื่อผู้ใช้จะถูกเก็บเป็นข้อความธรรมดา (`username`) ใน IndexedDB บนเครื่องของผู้ใช้ ส่วนรหัส PIN จะถูกเข้ารหัสแฮชด้วยอัลกอริทึม SHA-256 ผ่าน Web Crypto API ในเบราว์เซอร์ก่อนทำการบันทึกเป็น `pinHash` โดยไม่มีการส่งข้อมูลเหล่านี้ออกนอกเครื่องไปยังเซิร์ฟเวอร์ใด ๆ
- **ระบบป้องกันและล้างข้อมูลอัตโนมัติ (Auto-Wipe)**: หากป้อนข้อมูลผู้ใช้หรือรหัส PIN ผิดติดต่อกันรวมเป็นจำนวน **5 ครั้ง** ระบบจะทำการลบข้อมูลประวัติ คอลเลกชัน สภาพแวดล้อม และการตั้งค่าทั้งหมดที่บันทึกอยู่ในเบราว์เซอร์นี้ทันทีเพื่อรักษาความปลอดภัย จากนั้นจะรีเซ็ตแอปกลับสู่หน้าตั้งค่าเริ่มต้นใหม่
- **ปุ่มรีเซ็ตแบบแมนนวล (Forgot details)**: หากจำข้อมูลไม่ได้ พนักงานสามารถคลิก Forgot details? เพื่อยืนยันคำสั่งล้างข้อมูลในเครื่องทั้งหมดและเริ่มกระบวนการตั้งค่า Username และ PIN ใหม่ได้ด้วยตัวเอง

### 11.2 ระบบรักษาความปลอดภัยของ CORS Proxy (`api/proxy.ts`)
เมื่อรันระบบ CORS Proxy เพื่อยิง Request ไปยังเซิร์ฟเวอร์ที่ไม่รองรับ CORS หรืออยู่ในเครือข่ายเฉพาะ Proxy มีระบบตรวจสอบความปลอดภัยเสริมดังนี้:
- **การป้องกัน SSRF (Server-Side Request Forgery)**: ตัว Proxy จะทำการปฏิเสธคำขอที่พยายามเชื่อมต่อกลับมายัง Localhost (`localhost`, `127.0.0.1`, `::1`), ที่อยู่ IP ภายในองค์กร (Private IP ได้แก่ `10.x.x.x`, `192.168.x.x`, `172.16.x.x` - `172.31.x.x`) หรือ API เมทาดาตาของระบบคลาวด์ (`169.254.169.254`) เพื่อป้องการใช้ Proxy เป็นเครื่องมือสแกนหรือโจมตีระบบเครือข่ายภายใน
- **กำหนด Whitelist โดเมนที่อนุญาต (`ALLOWED_DOMAINS`)**: สามารถจำกัดให้ Proxy ยิง Request ไปยังโดเมนเฉพาะที่ได้รับการยอมรับผ่านตัวแปรสภาพแวดล้อม `ALLOWED_DOMAINS` (คั่นด้วยเครื่องหมายจุลภาค `,`)
- **การยืนยันตัวตนผ่าน Shared Secret (`PROXY_SECRET`)**: หากกำหนดตัวแปรสภาพแวดล้อม `PROXY_SECRET` ในระดับเซิร์ฟเวอร์ ทุกคำขอที่ส่งมาที่ Proxy จะต้องแนบเฮดเดอร์ `x-proxy-secret` ที่มีค่าตรงกัน มิฉะนั้นจะได้รับ HTTP 401 Unauthorized
- **การยืนยันตัวตนผ่าน JWT (`PROXY_JWT_SECRET`)**: หากต้องการตรวจสอบสิทธิ์พนักงานด้วยระบบ Identity Provider (IdP) ขององค์กร สามารถตั้งค่า `PROXY_JWT_SECRET` เพื่อให้ตัว Proxy ตรวจสอบลายเซ็น (HMAC SHA-256 Signature) ของ JWT Token ที่แนบมาในส่วนหัว `Authorization: Bearer <token>` ได้อย่างปลอดภัย
